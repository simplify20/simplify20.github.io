应用里怎么完成一件事：任务调度浅析

白阳 2020-12-06 concurrencyscheduler
生产流水线

引子

在移动应用开发过程中，大部分时候不需要关心所写的代码是如何执行的，比如说，我们定义好一个UI组件，程序运行时它的渲染逻辑就会被执行，而如何触发它的执行对我们来说是透明的。在浏览器环境中，web应用可能做UI展示、设置事件监听、进行IO操作等，即使在单线程的javascript运行环境下，上述行为也能被及时响应，而不会block用户。

无论Web，还是移动应用，各种逻辑执行行为都可以可以泛化为任务，而这些行为的表现取决于任务的执行机制及任务调度机制。

进程调度

再深入应用内的任务调度前，先放大视野，讲讲进程调度。在单任务操作系统中，每次只能运行一个程序，在多任务操作系统中，允许一到多个程序并发运行，并发运行是基于时间片(time slice)的：将CPU时间划分成一个个小段，然后将这些小段分配给进程。

图(LieBrother)

如图，多个进程共享时间片。时间片的分配基于各种进程调度算法：

进程调度算法

FIFO(First In First Out)

先进入队列的进程先运行，运行到完成或者阻塞时，再重新调度。这种算法一般会结合优先级进行调度

RR(Round Robin)

按照进程到达时间轮流调度。

  • 1.进程在给定时间执行结束，进入2，否则将进程移动到队尾，进入2
  • 2.下一个进程开始执行
  • 3.重复1 图片(wikipedia)

SPN(Shortest Job Next)

最短进程优先。将预期占用运行时间最短的进程优先执行，直到运行完成或阻塞时，再重新调度。

SRT(Shortest Remaining Time)

最短剩余时间优先。新进程进来时，如果新进程的预计运行时间比当前进程的剩余运行时间更短，就抢占当前进程

Android的进程调度

OOM_ADJ

  • oom_score_adj

Android的进程调度策略

Android基于Linux内核，在目前的Linux内核中，一共有六种调度策略，它们可以分为三类：

  • 普通调度策略：SCHED_NORMAL, SCHED_BATCH，SCHED_IDLE 。
  • 实时调度策略：SCHED_FIFO, SCHED_RR。
  • Deadline调度策略：SCHED_DEADLINE。

  • SCHED_NORMA 是进程默认调度策略

  • SCHED_BATCH 与SCHED_NORMAL类似，不同的是，内核会认为该进程是CPU密集型，因此在调度会有小的惩罚。这种策略适用于那些非交互的后台进程

  • SCHED_IDLE 优先级最低的调度策略

  • SCHED_FIFO/SCHED_RR 就是我们前面提到过FIFO和RR调度算法

  • SCHED_DEADLINE 指定了预计完成时间的调度策略，它拥有超过所有其他策略的最高优先级

总体优先级顺序：SCHED_DEADLINE > SCHED_FIFO = SCHED_RR > SCHED_NORMAL > SCHED_BATCH > SCHED_IDLE

cgroup

cgroup是control group的缩写。这是一个Linux内核的特性。用来对进程所使用的资源（如CPU、内存、磁盘输入输出等）进行限制、统计与隔离。

通过cgroup可以配置不同类型进程的cpusets

Android的进程调度总结

Android进程调度是基于优先级的，而这个优先级取决于Andrid系统对进程重要性的判定，系统进程是最核心的进程，是所有应用进程的依托，而应用进程的重要性，取决于用户的交互程度和感知度，这个是在设计交互应用时要考虑的，对于进程和线程的优先级，前台>后台，交互>非交互，感知>非感知，系统>应用。

此外，用户的交互行为，例如切换应用、启动应用，杀死应用，锁屏等，导致Android进程的状态也处于动态的变化中，需要根据不同进程状态设置不同的优先级，以快速响应用户或者节省资源。

关于进程调度的介绍就到此为止，更多内容可以参考Android系统上的进程管理：进程的调度,以及Android系统中的进程管理：进程的优先级 。这里有很多内容，参考了上面的文章，鸣谢！

接下来，我将会缩小视野，聊一聊进程内（应用内）的任务调度，而无论是进程调度还是进程内的任务调度，都是利用有限的计算资源，以最快的响应速度，并以一定策略完成一项或者多项工作。下面将介绍如何在应用中并行或串行地完成多项任务。

并发模型

设计模式指导我们如何设计高可扩展、高可维护的代码，而并发模型指导我们如何设计高效、安全的多任务执行器(调度器)。

并行工作器(Parallel Workers)

任务被提交到一系列并行工作的Worker中，这些worker工作在不同的线程(CPU)上，无法保证任务的执行顺序。java的ThreadPoolExecutor就使用了这个模型

图片Jenkov Aps

并行工作的Worker往往需要共享状态（内存或者数据库），ThreadPoolExecutor中不同Worker共享一个任务队列。当存在共享状态时，就可能存在竞态条件和死锁，程序也会变得复杂。Worker等待别的Worker完成共享状态的访问时，并行运行机制可能退化为串行工作机制。

图片Jenkov Aps

流水线工作器(Reactive,Event Driven Model)

一系列Worker分布在流水线上，这些Worker工作在不同线程，之间不共享内存，每个Worker只处理整个工作的一部分，Worker完成了自己的那一部分，将工作交给下个Worker，该模型的好处是可以实现有顺序的工作流程，不共享内存避免了并行工作器模型中存在的多线程资源竞争问题(race condition) 以及复杂性

图片Jenkov Aps

Actor模型(Event Driven Model)

在Actor模型中，每个Worker称之为Actor，Actor之间可以相互发消息，消息的发送和处理都是异步的，Actor模型可以处理一项或者多项工作

图片Jenkov Aps

Channel模型(Event Driven Model/CSP模型)

在Channel模型中，Worker之间不直接通信，而是通过向Channel发消息和接收消息通信，Worker之间是解耦的。golang协程和kotlin协程都采用了这种模型

图片Jenkov Aps

函数式并发模型(Functional Parallelism)

该模型认为一项工作可以由一系列函数调用完成，函数被视为Actor，一个函数调用另一个函数视为发送消息，传递到函数的参数是拷贝的，不存在被修改的情况，每个函数可以被独立地执行，java的ForkAndJoinPool实现了这种机制

图片permike

并发模型总结

生产流水线是在现实生活中可以看到的并发实例，不同的工人(机器) 分布于流水线的不同环节，完成各自既定的工序，如开篇图那样，每个工人完成自己的工序后，开始处理下一件产品，完成工序的产品通过传送带传到下一个工人，执行下一步工序。生产流水线对应到并发模型中就是流水线模型 ，每个工人之间存放正在传送的产品的空间可以称之为Buffer，有了这个Buffer，工人之间即使完成工序的速度不平衡，也不会出现一个工人处理不过来，一个工人没有产品可以处理的情况。

通过发消息发送任务请求，接收消息去处理任务的场景很多，例如分布式系统、Android中的Handler等，对应到并发模型，就是Actor模型

下面将介绍一种计算机软件中随处可见的任务执行模型：事件循环模型，可以看到，它是Actor模型的一种特例

事件循环（Event Loop）

除了并发式的任务执行，还有一种常见的执行方式称之为事件循环：

事件循环是一种程序设计结构，这种结构会等待消息，并分发消息给消息处理器去处理（wikipedia)

事件循环的伪代码如下：

function main
    initialize()
    while message != quit
        message := get_next_message()
        process_message(message)
    end while
end function

Android、iOS、Windows、javascript(浏览器)、Node.js主线程都采用了这种运行模式

Event Loop in Android

为便于理解，去掉了部分代码：

    public static void loop() {
        final Looper me = myLooper();
        final MessageQueue queue = me.mQueue;
        //1.循环
        for (; ; ) {
            //2.从消息队列取消息
            Message msg = queue.next(); // might block
            if (msg == null) {
                // 3.退出逻辑
                return;
            }
            //4.分发消息
            msg.target.dispatchMessage(msg);
        }
    }

消息分发： MessageQueue利用了linux的epoll机制：在消息队列为空或者消息没到执行时间时，释放CPU，避免不必要的资源浪费（忙等会浪费CPU)，从消息队列取消息的逻辑(为便于理解，去掉了部分代码)：

    Message next() {
        final long ptr = mPtr;
        for (; ; ) {
            //该方法可能会阻塞主线程
            // 1.nextPollTimeoutMillis>0：队列头部的消息，执行时间还没到，有限时阻塞，如果消息队列没有消息写入，nextPollTimeoutMillis时间后该方法返回，否则有消息写入时该方法返回
            // 2.nextPollTimeoutMillis<0：消息队列为空，无限时阻塞，有消息写入时该方法返回
            // 3.nextPollTimeoutMillis=0：不阻塞，立即返回
            nativePollOnce(ptr, nextPollTimeoutMillis);
            Message msg = mMessages;
            if (msg != null) {
                if (now < msg.when) {
                    // Next message is not ready.  Set a timeout to wake up when it is ready.
                    nextPollTimeoutMillis = (int) Math.min(msg.when - now, Integer.MAX_VALUE);
                } else {
                    // Got a message.
                    mBlocked = false;
                    if (prevMsg != null) {
                        prevMsg.next = msg.next;
                    } else {
                        mMessages = msg.next;
                    }
                    msg.next = null;
                    if (DEBUG) {
                        Log.v(TAG, "Returning message: " + msg);
                    }
                    msg.markInUse();
                    return msg;
                }
            } else {
                // No more messages.
                nextPollTimeoutMillis = -1;
            }
        }
    }

Event Loop总结

Event Loop模型多用于UI应用中，UI应用需要不断地响应用户的输入事件，以一定频率不断刷新UI界面，这些操作可以通过Event Loop完成，当用户输入事件发生或者UI刷新信号到达时，向消息队列中添加相应的消息，Event Loop取出消息，执行相应回调或者执行UI渲染指令即可。

不同任务对硬件有不同的需要，按照任务对CPU、IO速度要求的不同可以分为CPU密集型和IO密集型

CPU密集型、IO密集型（CPU-bound/IO-bound）

  • 如果CPU更快或者有更多的CPU，任务执行的会更快，这样的任务是CPU密集型任务，任务主要的时间在使用CPU（做计算）

    • 视频编解码、圆周率计算、CG渲染等

  • 如果IO速度更快，任务执行的会更快，这样的任务是IO密集型任务，任务主要的时间在等IO

    • 网络请求、磁盘读写等

了解任务的分类后，我们知道，如果将一个IO密集型的任务和一个CPU密集型的任务放到流水线模型中会存在这样的问题：IO密集型任务等待IO的过程中，下一个环节的CPU密集型任务无法执行。下面将介绍的IO模型将解答这个问题。

IO模型

阻塞IO(BIO)

IO操作后，内核返回结果前应用程序处于阻塞状态

非阻塞IO(NIO)

IO操作后，内核马上返回结果，需要轮询查IO结果

多路复用IO

同时等待多个i/o通道的数据：select/poll/epoll

epoll相较于select/poll，多了两次系统调用，其中epoll_create建立与内核的连接，epoll_ctl注册事件，epoll_wait阻塞用户进程，等待IO事件。

异步IO(AIO)

用户进程发出IO系统调用后立即返回，内核等待数据准备完成，然后将数据拷贝到用户进程缓冲区，然后发送信号告诉用户进程IO操作执行完毕

同步IO

阻塞IO、非阻塞IO、多路复用IO均为同步IO

IO模型总结

javascript是单线程的，如果执行过程涉及很多同步IO操作是否就会存在以下问题：

上图的绿色部分是程序的运行时间，红色部分是等待时间。从事实上看并不会这样。为了解决上述问题，javascript使用了上面提到过的Event Loop程序执行机制

上图中，主线程运行的时候，产生堆（heap）和栈（stack），栈中的代码调用各种外部API，它们在"任务队列"中加入各种事件（click，load，done）。只要栈中的代码执行完毕，主线程就会去读取"任务队列"，依次执行那些事件所对应的回调函数。javascript中，同步IO操作被放到单独的线程中执行，执行完毕后向任务队列中加入相应的回调，主线程读取到任务队列中相应的回调，并执行，一次IO结束，这样就避免了同步IO对主线程的阻塞问题。

总结

本篇文章以”应用中如何完成一件事“为题，标题比较宽泛，本质上讲，每行代码的执行都是应用在完成一件事，我所指的”事“侧重于任务(task)，或者一个代码执行单元。首先从进程这个”大任务“出发，视野逐渐缩小，逐渐深入到一个callback、runnable、microtask的执行。无论是前者还是后者，运行时都需要cpu和内存资源。

在进程视野，操作系统会将进程分类，并使用不同的调度策略和优先级，操作系统在保证系统服务运行资源足够的前提下，与用户有交互行为的进程在资源分配上将会侧重。操作系统是面向用户，为用户提供服务的，这种侧重无疑体现了这一点。

再缩小到应用视野，每个应用至少有一个进程与之对应，每个进程可能运行着一到多个线程，为了充分利用CPU资源，实现多个任务的并发，需要根据业务场景，结合并发模式设计不同的并发程序，以达到最大的任务吞吐量，同时避免不必要的复杂度和资源浪费，并行工作器在java中被广泛的使用，Actor模型中的特例事件循环模型在UI应用和C/S架构中被广泛使用。

和后台应用不同，UI应用大部分时间都是在主线程做事，使用多线程一般出于以下几个目的：

  • 优化资源利用：等IO时，CPU是空闲状态，可以做更多事
  • 一些情况下简化程序设计：读和处理分开
  • 快速响应：不阻塞UI线程

golang协程，在操作系统进程（线程）调度机制的基础上，进行了扩展，实现了更强的多任务执行能力

图 陈Qiu凯：Go Runtime的M-P-G模型

  • M：Machine，对应OS线程的概念，OS线程是CPU的最小调度单元，由内核调度模块负责调度。一般Runtime会限制同时运行M（阻塞的M另算）的数量与CPU核数（逻辑核）相同，这样可以减少OS级别的上下文切换，大部分都切换都在用户态完成。
  • P：Process，虚拟处理器的概念，一般一个逻辑核分配一个P，负责本逻辑核上的goroutine调度，解决全局共享队列的锁竞争问题。M与P互相关联，M不能独立于P存在。
  • G：Goroutine，用户态协程，执行用户代码的实体，维护了堆栈与状态相关的信息，并分配了很小的栈空间（通常2KB）。

Go Runtime的调度模式：

  • Go Runtime有两种Work队列：LRQ(local runnable queue)和GRQ(global runnable queue),并利用了Work-Stealing策略，在某一LRQ为空时从其他LRQ或者GRQ中取一半的任务执行
  • 当陷入系统调用或者同步IO调用时，M与P会暂时解绑，P会创建一个新的M,即创建一个新的线程，将新到达的G分配给新M执行
  • 当进入异步IO调用，阻塞的G会进入network poller,而M则继续服务其他的G，在Linux平台，network poller实际上是对epoll的封装，注册阻塞的fd并监听到达的事件，就绪的fd所在的G就会被重新调度，回到原来的LRQ等待被执行。当异步IO有结果时，阻塞的G重新进入LRQ

这就是Go Runtime的高效调度模型，可以做到“无间断执行代码”！

后续在开发中结合这些知识，可以针对具体业务场景，选择合理的调度策略和优先级，毕竟操作系统资源有限，每个进程都是操作系统的公民，nice值应该高一点。在一些场景中，如果现有的库或框架无法满足需求，可以尝试自定义任务调度器（执行器）。
